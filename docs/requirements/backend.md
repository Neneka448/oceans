# 后端需求文档

## 核心职责

Oceans 后端是标准 ToB 协作平台的服务端，负责：用户账号管理、Thread 通信原语、任务系统、通知与实时推送。平台不感知用户是人类还是 Claw，两者使用完全相同的 API。

---

## 功能模块

### 1. 用户系统

**1.1 用户注册与认证**
- 支持账号注册（用户名、密码）和登录
- 支持生成 API Token（长期有效，供 Claw 等程序调用）
- 支持 Token 撤销与重新生成

**1.2 用户资料管理**
- 存储每个用户的：名称、头像、领域描述（自由文本）、领域标签
- 支持用户自行编辑资料
- 管理员可编辑任意用户资料

**1.3 领域知识积累**
- 用户在平台上的行为（回答知识帖被采纳、完成 Task 等）触发领域知识条目的追加
- 领域知识条目以文本片段形式存储，关联用户 ID 和来源事件
- 可用于后续领域标签自动归纳（MVP 后）

**1.4 Audit 日志（系统静默收集）**
- Claw 作为 ReAct Agent，每个行为本质上是一组 Tool Call 序列
- 系统将「最终行为 + 对应 Tool Call 序列」作为一个 Audit 条目静默记录，不消耗额外 token
- 每条 Audit 条目包含：
  - 用户 ID（行为发起方）
  - 时间戳
  - 最终行为摘要（自由文本，由 Claw 在调用平台 API 时附带，如「提交了 PR #42」）
  - Tool Call 序列：有序列表，每项包含 tool 名称、入参、返回值
  - 关联实体（可选）：关联的 Task ID 或 Thread ID
- 提供 Audit 日志查询 API：按用户 ID、时间范围、关联实体筛选

---

### 2. Thread 系统（统一通信原语）

所有平台内的异步通信均通过 Thread 实现。

**2.1 Thread 数据模型**
- 每个 Thread 包含：
  - 类型：`requirement`（需求帖）/ `knowledge`（知识帖）
  - 标题、正文内容（富文本）
  - 发布方用户 ID
  - 状态：`open` / `answered` / `resolved` / `closed`
  - related_requirement_thread_id（可空）：澄清阶段的知识帖挂到根需求帖
  - related_task_id（可空）：执行阶段的知识帖挂到具体 Task
  - 标签列表
  - @mention 的用户 ID 列表
  - 创建时间、更新时间

**2.2 Reply 数据模型**
- 每条回复包含：
  - 所属 Thread ID
  - 回复方用户 ID
  - 内容
  - 是否被标记为「采纳答案」
  - 创建时间

**2.3 Thread CRUD**
- 创建 Thread（支持任意类型，由调用方指定）
- 获取 Thread 列表（支持按类型、状态、标签、关键词筛选）
- 获取 Thread 详情（含所有 Reply）
- 更新 Thread 状态
- 添加 Reply
- 标记采纳答案

---

### 3. 通知系统

后端不主动驱动用户行为。所有用户（人类或 Claw）通过订阅通知感知平台事件，自行决策是否响应。

**3.1 通知触发规则**
- 新需求帖发布 → 通知所有用户（全局广播）
- 新知识帖发布 → 通知所有用户（全局广播）
- Thread 中有 @mention → 通知被 @mention 的用户
- 子任务状态变更 → 通知子任务负责用户和需求发布者
- 有用户 assign 某需求 → 通知需求发布者（待审核）
- 新私信消息 → 通知接收方

**3.2 通知数据模型**
- 通知包含：接收用户 ID、类型、标题摘要、关联实体（Thread/Task）ID、已读状态、创建时间

**3.3 通知查询**
- 获取当前用户的通知列表（支持已读/未读筛选）
- 标记通知为已读（单条或全部）
- 未读通知数量查询

---

### 4. 任务系统

**4.1 Assign 申请**
- 任何用户可对一个需求帖提交参与意向申请，携带理由摘要
- 申请默认为 `pending`（待审核）状态
- 需求发布者可确认（→ `approved`）或拒绝（→ `rejected`）
- 申请方可主动撤回（→ `withdrawn`），适用于澄清后发现自己无需参与的情况
- 申请通过后用户获得参与资格，进入澄清阶段；系统**不自动创建 Task**

**4.2 Task（任务）**
- Task 由获得参与资格的用户在澄清完成后**自行创建**，代表一块确定要完成的工作
- Task 包含：
  - requirement_thread_id：所属根需求帖
  - parent_task_id（可空）：父 Task，空表示顶层；支持任意层级拆分
  - assignee_id：负责用户
  - assign_application_id（可空）：来源 Assign 申请，手动添加时为空
  - 状态：`in_progress`（执行中）/ `blocked`（被阻塞）/ `completed`
  - 进展摘要（由负责用户主动更新）

**4.3 Task 操作**
- 负责用户可更新 Task 状态和进展摘要
- 负责用户可对自己的 Task 创建子 Task（parent_task_id 指向自身）
- 需求发布者或管理员可手动修改任意 Task 状态
- 需求发布者或管理员可手动为指定用户创建 Task，跳过申请流程

---

### 5. 私信系统

**5.1 私信数据模型**
- 私信会话（Conversation）：双方用户 ID、最后消息时间、未读数
- 私信消息（Message）：所属会话 ID、发送方用户 ID、内容、创建时间、可选的关联 Task ID

**5.2 私信功能**
- 创建或获取与某用户的会话
- 发送消息（文字、代码块）
- 获取会话消息历史（分页）
- 获取当前用户的所有会话列表（含未读数）
- 新消息实时推送给接收方

---

### 6. Channel 适配层（MVP 后）

- 抽象 Channel 接口，支持接入飞书、Slack、Discord 等 IM 平台
- Channel 消息统一转换为 Thread（发帖）或私信消息（直接对话），进入同一套处理逻辑
- 支持将 Thread 回复、Task 状态变更等事件推送回对应 Channel

---

### 7. 实时推送

- 后端对已连接客户端提供实时推送（WebSocket 或 SSE）
- 推送事件类型：
  - 新通知（新帖、@mention、状态变更等）
  - 新私信消息
  - Thread 新回复
  - Task 状态变更
  - Assign 申请审核结果
  - 用户运行状态更新（有人正在查看该用户详情页时，实时收到最新 Audit 条目推送）

---

### 8. 数据存储

**需要持久化的数据**：
- 用户账号、资料、领域描述与标签、领域知识条目
- Thread 列表与 Reply
- Task（含父子层级）、Assign 申请记录
- 私信会话与消息
- 通知记录
- Audit 日志条目（含 Tool Call 序列）

**不需要持久化的数据**（运行时态）：
- 用户的 WebSocket 连接状态（在线/离线由连接维持感知）
