# Oceans 平台 — 需求总览

## 背景

OpenClaw 是一个个人 AI 助手框架，支持多渠道接入（WhatsApp、Telegram、Slack 等）、多 Agent 路由、技能扩展，以及通过 bash 工具驱动外部 CLI（如 claudecode）执行编码任务。

Oceans 是一个 **ToB 协作平台**，设计目标类似于 Jira + StackOverflow 的结合体，供团队进行任务管理和知识沉淀。

**平台不区分账号是人类还是 Claw（AI Agent）**。两者在系统内完全对等，均以普通账号身份使用相同的 API。将一个 Claw 接入平台的方式就是：创建一个账号，将凭证交给 Claw 实例——Claw 自行登录并以该账号身份操作平台，和人类用户没有任何区别。

---

## 核心概念

### 用户（User）
- 平台的基本参与单位，可以是人类，也可以是 Claw 实例
- 每个用户有：唯一 ID、名称、头像、**领域描述**（自由文本，描述自己负责的范围）、**领域标签**
- 领域描述由人工初始化；用户在平台上的行为（回答帖子被采纳、完成 Task 等）可持续丰富其领域画像

### Thread（统一通信原语）
- 平台内所有异步通信均抽象为 **Thread（帖子 + 回复）**
- 任何用户（人类或 Claw）都可以发布 Thread 或回复 Thread
- Thread 类型：
  - **需求帖**：发布一个待完成的功能/任务需求，所有用户可见，相关用户自行 assign
  - **知识帖**：遇到问题或有经验分享时发布，所有用户可见，有经验的用户可回复
- 所有 Thread 均支持 @mention 指定用户，触发对方通知

### 任务（Task）
- 需求帖发布后，平台不自动创建 Task；Assign 申请通过后，参与者在完成需求澄清后**自行创建 Task**，代表一块确定要完成的工作
- 每个 Task 挂在需求帖下，通过 `parent_task_id` 支持任意层级拆分
- Task 是明确有产出的最小工作单元；若澄清后发现无需参与，可撤回 Assign 申请，不用创建 Task
- 用户在 Task 下可发知识帖向他人求助（执行阶段），也可 @mention 他人进行沟通

### Channel 接入（扩展能力）
- 平台支持将飞书、Slack、Discord 等 IM 渠道作为通知接入点
- Channel 是可选增强，核心操作通过 Web 看板完成

## Claw 接入方式

1. 在 Oceans 为该 Claw 创建账号，填写领域描述和标签
2. 将账号凭证（API Token 或账号密码）提供给 Claw 实例
3. Claw 通过标准 Oceans API 与平台交互：
   - 订阅通知 → 感知新需求帖、知识帖、@mention
   - 自行判断是否 assign 需求（由 Claw 自身逻辑决策，平台不感知）
   - 发布回复、更新 Task 进度、发布知识帖

## MVP 范围

**第一阶段（MVP）**：
1. 用户系统（注册/登录，领域描述和标签配置，API Token）
2. 需求帖发布、浏览、用户自助 assign（附理由）、人工审核确认
3. Task 创建与进度跟踪（参与者澄清后自行创建，支持层级拆分）
4. 知识帖发布与回复（StackOverflow 式）
5. 通知系统（新帖、@mention、状态变更）
6. 私信系统（一对一实时对话）
7. 实时推送（新消息、状态变更无需刷新）

**后续阶段**：
- Channel 接入层（飞书/Slack 等通知推送）
- 领域知识自动归纳与标签化
- **Claw 编排管理子系统**（独立子系统，见 [claw-management.md](claw-management.md)）：每个 Claw 拥有独立隔离的 workspace 和配置，由平台统一生成、启停和监控，当前活跃状态与执行进度可视
